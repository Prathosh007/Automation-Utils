UEMS Agent HTTPS Traffic Capture & Validation Framework

A Python-based MITM proxy automation framework for intercepting, capturing, and validating UEMS agent HTTPS communication using mitmproxy. Follows the JSON-driven, GOAT-server-compatible pattern used across Agent Automation projects.

1. What It Does

This tool acts as a man-in-the-middle proxy between the UEMS agent and the DC server. It intercepts all HTTPS traffic the agent sends, captures request/response data, and runs automated validation test cases against the captured traffic.

UEMS Agent → mitmproxy (127.0.0.1:8080) → Real DC Server (HTTPS) | Captures all traffic (URLs, headers, bodies, status codes)

Key Capabilities

  • Traffic Recording — Capture all agent HTTPS communication indefinitely until stopped
  • Request Validation — Verify HTTP method, URL parameters, headers, body content
  • Response Validation — Verify status codes, response headers, body content
  • Request Count Assertion — Verify expected number of requests per endpoint
  • Request Order Assertion — Verify correct request sequence
  • Traffic Export — Dump captured data to JSON for analysis

2. How It Works

Step-by-step execution flow:
  1. Install mitmproxy CA certificate into Windows trust store
  2. Patch Windows Registry to redirect agent traffic to local proxy
  3. Apply OS-level traffic lock (hosts file + firewall + port proxy) to prevent agent bypass
  4. Start mitmproxy as a reverse proxy forwarding to the real server
  5. Trigger cfgupdate.exe to force agent config refresh
  6. Capture and/or validate all HTTPS traffic
  7. Save captured traffic to JSON report
  8. Cleanup: restore all OS-level changes, restore registry, stop proxy

Why OS-Level Traffic Lock?

The agent reads the real server address from the meta-data.xml response body and can reconnect directly, bypassing the proxy. Three OS-level locks prevent this:

LayerMechanismWhat It Blocks
Hosts File127.0.0.1 <server-hostname>Hostname-based connections
Port Proxy127.0.0.1:8383 → 127.0.0.1:8080Agent using port read from XML
FirewallBlock outbound to real server IPDirect IP-based connections

All three are applied before capture and removed automatically during cleanup.

3. JSON-Driven Test Engine

All test scenarios are defined as JSON files. Each file contains an array of commands executed sequentially. No code changes are needed to add new test cases — just write a new JSON file.

[
  {"id": "STEP_001", "action": "start_proxy", "listen_port": 8080},
  {"id": "STEP_002", "action": "wait_for_request", "url_pattern": "agentSlot", "timeout": 180},
  {"id": "TC_001",   "action": "validate_request", "url_pattern": "agentSlot", "method": "GET"},
  {"id": "TC_002",   "action": "validate_response", "url_pattern": "agentSlot", "expected_status": 200},
  {"id": "STEP_003", "action": "stop_proxy"}
]

16 built-in actions are available: proxy control, certificate management, registry patching, OS commands, traffic validation, response modification, and more.

5. Test Cases — Example JSON

5.1 Request Validation

Verify what the agent sends — HTTP method, URL parameters, headers, body content.

// Verify agentSlot request is GET
{"id": "TC_001", "action": "validate_request",
 "url_pattern": "agentSlot", "method": "GET",
 "description": "agentSlot: method is GET"}

// Verify Authorization header has Basic auth
{"id": "TC_002", "action": "validate_request",
 "url_pattern": "agentSlot",
 "header_name": "Authorization", "expected_value": "Basic ",
 "validation_type": "startswith",
 "description": "agentSlot: has Basic auth"}

// Verify URL contains expected query parameter
{"id": "TC_003", "action": "validate_request",
 "url_pattern": "agentSlot",
 "expected_value": "actionToCall=AgentOfset",
 "validation_type": "contains",
 "description": "agentSlot: URL has actionToCall param"}

// Verify POST body contains expected field
{"id": "TC_004", "action": "validate_request",
 "url_pattern": "endpointdlp",
 "body_pattern": "ResourceId",
 "validation_type": "contains",
 "description": "endpointdlp: body has ResourceId"}

// Verify POST method on application control endpoint
{"id": "TC_005", "action": "validate_request",
 "url_pattern": "appctrl-new-storeapp-scan-data", "method": "POST",
 "description": "appctrl store apps: method is POST"}

6. How to Run

Run with Python

:: Run as Administrator
.venv\Scripts\python.exe main.py TestData\general_validation.json --show-all

Run with EXE

:: Run as Administrator
CommunicationAutomation.exe TestData\general_validation.json --show-all

--show-all prints every step result. Without it, only failures and the summary are shown.

8. Future Scope — Data Interruption & Failure Simulation

The framework already has modify_response and block_request actions built in. These enable data interruption — altering or blocking live traffic to simulate failure scenarios and test how the agent handles them. Below are planned use cases and ideas.

8.1 Already Available (Ready to Use)

ActionWhat It DoesJSON Example
modify_response Replace response status, body, or headers for matching requests {"action": "modify_response", "url_pattern": "meta-data\\.xml", "inject_status": 500, "inject_body": "Internal Server Error"}
block_request Completely block or delay matching requests {"action": "block_request", "url_pattern": "agentSlot", "block_action": "block"}
block_request (delay) Add artificial latency to simulate slow network {"action": "block_request", "url_pattern": ".*", "block_action": "delay", "delay_ms": 5000}

Rules can be "persistent": true (apply to every matching request) or one-shot (apply once then auto-remove).
