CommunicationAutomation — Detailed Architecture

1. System Overview

CommunicationAutomation is a JSON-driven test automation framework that uses a man-in-the-middle (MITM) proxy to intercept, capture, and validate UEMS agent HTTPS communication. It is built in Python and uses mitmproxy as the proxy engine.

+------------------+ +------------------+ +------------------+ | | HTTPS | | HTTPS | | | UEMS Agent | -------> | mitmproxy | -------> | DC Server | | (on machine) | | (127.0.0.1: | | (remote) | | | <------- | 8080) | <------- | | +------------------+ +--------+---------+ +------------------+ | | captures every | request/response v +--------+---------+ | | | TrafficStore | | (in-memory) | | | +--------+---------+ | | queried by v +--------+---------+ | | | Test Engine | | (validates, | | asserts, | | reports) | | | +------------------+

4. Component Details

4.1 TestEngine (core/test_engine.py)

The orchestrator. Loads JSON, iterates commands, dispatches to actions, collects results.

ResponsibilityHow
Load test definitionReads JSON file, supports flat array [...] or wrapped {"commands": [...]}
Variable substitutionCalls VariableContext.substitute_command() before parsing each command
Command dispatchLooks up action handler from ActionRegistry, calls action.run()
Flow controlStops on first failure unless continueOnFailure: true
Shared contextPasses context dict to all actions — holds proxy_manager, traffic_store, rule_engine
Result trackingAppends every CommandResult to TestResult.command_results[]
OutputPrints GOAT-compatible SUMMARY|Total:N|Passed:N|Failed:N

4.2 Command Model (core/models.py)

A single dataclass that holds ALL possible fields for every action type. The from_dict() class method maps camelCase JSON keys to snake_case Python attributes.

Why one big dataclass? Keeps the JSON format flat and simple. Each action reads only the fields it needs and ignores the rest. No inheritance hierarchy, no polymorphic deserialization needed.

Key field groups:

4.3 ActionRegistry (core/action_registry.py)

A simple dictionary mapping action name strings to singleton handler instances.

ActionRegistry._actions = { "start_proxy" -> StartProxyAction() "stop_proxy" -> StopProxyAction() "install_certificate" -> InstallCertificateAction() "patch_registry" -> PatchRegistryAction() "run_command" -> RunCommandAction() "wait_for_request" -> WaitForRequestAction() "wait_for_interrupt" -> WaitForInterruptAction() "validate_request" -> ValidateRequestAction() "validate_response" -> ValidateResponseAction() "modify_response" -> ModifyResponseAction() "block_request" -> BlockRequestAction() "assert_request_count" -> AssertRequestCountAction() "assert_request_order" -> AssertRequestOrderAction() "capture_traffic" -> CaptureTrafficAction() "clear_captures" -> ClearCapturesAction() ... (18 total) } get_action("validate_request") -> returns ValidateRequestAction instance
Extensibility: To add a new action, create a new class in actions/, extend BaseAction, implement execute(), and register it in ActionRegistry.__init__(). No other code changes needed.

4.4 BaseAction (core/base_action.py)

Abstract base class using the Template Method design pattern.

BaseAction.run(command, context) <-- Template Method | +-- validate_command(command) <-- Override in subclass | returns True/False | +-- log_execution(command) <-- Logs to file | +-- start timer | +-- execute(command, context) <-- ABSTRACT: subclass implements | | | +-- handle_success(command, result, remarks) | | - result.success = True | | - Logger.output_remark(id, True, remarks) | | | +-- handle_failure(command, result, error) | - result.success = False | - Logger.output_remark(id, False, error) | +-- stop timer | +-- return CommandResult

Every action only needs to implement execute(). Timing, logging, error handling, and output formatting are handled by the base class.

5. Proxy Layer — How Traffic Capture Works

5.1 ProxyManager (proxy/proxy_manager.py)

Manages the mitmproxy DumpMaster lifecycle.

Main Thread Daemon Thread (mitmproxy-thread) ----------- --------------------------------- ProxyManager.start() | +-- create threading.Event() +-- start daemon thread ---------> _run_proxy() +-- wait for Event (15s timeout) | | +-- asyncio.run(_async_run_proxy()) | | | +-- create Options(mode, port, ...) | +-- create DumpMaster(opts) | +-- add UEMSAddon (captures traffic) | +-- add _StartupNotifier | | | | | +-- running() hook fires | Event.set() <-------------------------- | -> sets Event | | +-- proxy is ready! +-- await master.run() (blocks until shutdown) ProxyManager.stop() | +-- master.shutdown() ----------------> DumpMaster exits run() +-- thread.join(10s) Thread ends
Key design decision: DumpMaster must be created inside an async context because it calls asyncio.get_running_loop() internally. That's why we use asyncio.run() with an async helper method, not just a plain thread target.

5.2 UEMSAddon (proxy/addon.py)

The mitmproxy addon that hooks into every HTTP flow.

Incoming Request | v UEMSAddon.request(flow) | +-- Check ResponseRuleEngine for matching rule | | | +-- rule.block = True? | | -> Return 502 "Blocked by UEMS proxy automation" | | -> flow.response set, request never reaches server | | | +-- rule.delay_ms > 0? | | -> time.sleep(delay_ms / 1000) | | | +-- rule.inject_status or inject_body? | -> Create synthetic Response (status, body, headers) | -> flow.response set, request never reaches server | +-- No rule / delay only -> request forwarded to real server | v Server responds (or synthetic response was injected) | v UEMSAddon.response(flow) | +-- _capture_flow(flow) -> Extract URL, method, headers, body from request -> Extract status, headers, body from response -> Create CapturedFlow dataclass -> TrafficStore.add_flow(captured) If upstream connection fails (e.g. TLS error): | v UEMSAddon.error(flow) | +-- _capture_flow(flow, is_error=True) -> Still captures the request data -> Response body = "<proxy error: ...>"

5.3 TrafficStore (proxy/traffic_store.py)

Thread-safe in-memory storage for captured flows with a wait/notify pattern.

TrafficStore +------------------------------------------+ | _lock: threading.Lock | | _flows: list[CapturedFlow] | <-- All captured traffic | _waiters: list[(pattern, method, Event)]| <-- Pending wait_for_request +------------------------------------------+ add_flow(captured): 1. Acquire lock 2. Append to _flows 3. For each waiter: if pattern matches -> event.set() 4. Remove notified waiters 5. Release lock wait_for_request(pattern, timeout): 1. Acquire lock 2. Check existing flows (reversed) for match - Found -> return immediately 3. Register waiter: (pattern, method, Event) 4. Release lock 5. event.wait(timeout) <-- blocks calling thread 6. If event set -> find and return matching flow If timeout -> return None get_latest_flow(pattern): - Iterate _flows reversed, return first match count(pattern): - Count matching flows get_all_flows(): - Return copy of all flows
Wait/Notify pattern: The wait_for_request action blocks the test engine thread until a matching request arrives on the proxy thread. This is how the test waits for the agent to make specific requests without polling.

5.4 ResponseRuleEngine (proxy/response_rules.py)

Manages rules for blocking, delaying, or modifying responses.

ResponseRuleEngine +-------------------------------------------+ | _rules: list[ResponseRule] | +-------------------------------------------+ | add_rule(rule) | <-- Called by modify_response / block_request actions | get_matching_rule(url, method) | <-- Called by UEMSAddon.request() | - returns first matching rule | | - one-shot rules removed after firing | | - persistent rules stay | | clear_rules() | +-------------------------------------------+ ResponseRule: url_pattern - regex to match URL method - HTTP method filter inject_status - replace status code (e.g. 500) inject_body - replace response body inject_headers- replace/add response headers block - block request entirely (502) delay_ms - add artificial latency persistent - true=apply to all matches, false=one-shot

6. Utils Layer

6.1 Logger (utils/logger.py)

Dual output: GOAT-compatible stdout + timestamped log file.

STDOUT (GOAT format): TC_001|PASSED|agentSlot: method is GET TC_002|FAILED|Failed: Expected status 200, got 401 SUMMARY|Total:10|Passed:8|Failed:2 Log File (Logs/execution-20260223_143022.log): [2026-02-23 14:30:22.123] Logger initialized [2026-02-23 14:30:22.456] EXECUTING: start_proxy - Start reverse proxy [2026-02-23 14:30:23.789] SUCCESS: SETUP_003 - Proxy started on 127.0.0.1:8080 [2026-02-23 14:30:45.012] FAILED: TC_002 - Expected status 200, got 401

--show-all flag: By default, only FAILED lines print to stdout. With --show-all, PASSED lines also print.

6.2 VariableContext (utils/variable_context.py)

Enables cross-step data passing using {{stepId.property}} placeholders.

Step SETUP_003 completes -> VariableContext stores: SETUP_003.success = "True" SETUP_003.data = (action-specific data) SETUP_003.message = "Proxy started on 127.0.0.1:8080" SETUP_003.duration = "0:00:01.234" Later step can reference: {"url_pattern": "{{SETUP_003.data}}"} -> substituted before Command.from_dict()

6.3 PatternMatcher (utils/pattern_matcher.py)

Two functions used across the entire framework:

FunctionUsed ByPurpose
matches_pattern(text, regex)TrafficStore, ResponseRuleEngine, AssertRequestOrderCheck if URL matches a regex pattern
validate_value(actual, expected, type)ValidateRequest, ValidateResponseCompare values using 7 validation types: contains, exact, startswith, endswith, regex, notempty, isempty

7. Shared Context — How Components Connect

The context dict is the glue between all actions. It flows through every action's execute(command, context) call.

context = {} (created empty by TestEngine) | v StartProxyAction.execute(): - Creates ProxyManager - ProxyManager creates TrafficStore + ResponseRuleEngine - context["proxy_manager"] = proxy_manager - context["traffic_store"] = proxy_manager.traffic_store - context["rule_engine"] = proxy_manager.rule_engine | v WaitForRequestAction.execute(): - store = context["traffic_store"] - store.wait_for_request("agentSlot", timeout=180) | v ValidateRequestAction.execute(): - store = context["traffic_store"] - flow = store.get_latest_flow("agentSlot") - validate flow.request_method == "GET" | v ModifyResponseAction.execute(): - engine = context["rule_engine"] - engine.add_rule(ResponseRule(url="meta-data", inject_status=500)) | v CaptureTrafficAction.execute(): - store = context["traffic_store"] - flows = store.get_all_flows() - write to JSON file | v StopProxyAction.execute(): - pm = context["proxy_manager"] - pm.stop()

8. Threading Model

+----------------------------------------------------------+ | MAIN THREAD | | | | main.py -> TestEngine -> Actions (sequential) | | | | | +-- wait_for_request: blocks on Event.wait() | | +-- wait_for_interrupt: blocks on stop_event.wait() | | +-- validate_*: reads from TrafficStore | +---------------------------+------------------------------+ | context dict (shared) | +---------------------------+------------------------------+ | DAEMON THREAD (mitmproxy-thread) | | | | asyncio event loop | | +-- DumpMaster.run() | | +-- UEMSAddon.request() -> check rules | | +-- UEMSAddon.response() -> TrafficStore.add_flow() | | +-- UEMSAddon.error() -> TrafficStore.add_flow() | +----------------------------------------------------------+ Thread safety: - TrafficStore: protected by threading.Lock - ResponseRuleEngine: protected by threading.Lock - Event objects: thread-safe by design - context dict: only written by start_proxy (before other actions read it)

10. Complete Data Flow — End to End

1. User runs: CommunicationAutomation.exe test.json --show-all 2. main.py reads test.json [ {"id":"S1", "action":"start_proxy", "mode":"reverse:https://172.24.148.221:8383"}, {"id":"W1", "action":"wait_for_request", "url_pattern":"agentSlot", "timeout":180}, {"id":"T1", "action":"validate_request", "url_pattern":"agentSlot", "method":"GET"}, {"id":"T2", "action":"validate_response", "url_pattern":"agentSlot", "expected_status":200}, {"id":"S2", "action":"stop_proxy"} ] 3. S1: StartProxyAction - Creates ProxyManager -> DumpMaster on 127.0.0.1:8080 - Daemon thread starts, proxy listening - context["traffic_store"] = TrafficStore (empty) 4. Meanwhile, UEMS agent sends HTTPS to 127.0.0.1:8080: GET /agentSlot?actionToCall=AgentOfset&computerName=... | +-> mitmproxy forwards to 172.24.148.221:8383 +-> Server replies 200 OK with XML body +-> UEMSAddon.response() captures: CapturedFlow(url="/agentSlot?...", method="GET", status=200, body="...") +-> TrafficStore.add_flow() stores it + notifies waiters 5. W1: WaitForRequestAction - Calls traffic_store.wait_for_request("agentSlot", timeout=180) - TrafficStore finds matching flow (already captured) - Returns immediately 6. 11. Design Patterns Used

PatternWhereWhy
Template MethodBaseAction.run()Common flow (validate, time, execute, handle errors) with action-specific execute() override
Registry / StrategyActionRegistryMaps action name strings to interchangeable handler objects at runtime
Observer (Wait/Notify)TrafficStoreTest thread waits for proxy thread to capture matching traffic, without polling
Singleton per ActionActionRegistry.__init__Each action handler created once, reused across all commands
Shared Contextcontext dictDependency injection without a DI framework — actions share proxy_manager, traffic_store, rule_engine
Data-Driven TestingJSON test filesTest logic defined in data (JSON), not code. New tests = new JSON, no recompilation.
Chain of ResponsibilityResponseRuleEngineRules checked in order, first match wins, one-shot rules auto-remove
